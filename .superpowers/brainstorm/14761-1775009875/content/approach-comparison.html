Architecture Approach

Three.js + TypeScript 기반 샌프란시스코 오픈월드 아키텍처 3가지 접근법

A

Layered Module (레이어드 모듈)

현재 코드를 기능별 모듈로 분리. 클래식 레이어 아키텍처.

src/
  core/ ← 렌더러, 씬, 카메라, 루프
  world/ ← 지형, 물, 하늘, 청크로더
  landmarks/ ← 금문교, 트랜스아메리카 등
  city/ ← 절차적 건물, 도로, 나무
  vehicles/ ← 차량, 보트, 갈매기
  camera/ ← 비행 컨트롤, 시네마틱 경로
  atmosphere/ ← 낮밤, 날씨, 포스트프로세싱
  ui/ ← HUD, 미니맵, 로딩

Pros

  • 현재 코드에서 자연스럽게 전환
  • 이해하기 쉬운 구조
  • 각 모듈 독립적으로 개발/테스트

Cons

  • 모듈 간 의존성이 복잡해질 수 있음
  • 수천 엔티티 관리시 성능 병목
  • 날씨↔재질 등 크로스커팅 관심사 처리 어려움
B

ECS + Systems (엔티티 컴포넌트 시스템)

bitECS 기반 데이터 지향 설계. 모든 오브젝트가 엔티티+컴포넌트.

src/
  ecs/ ← World, 엔티티 생성, 쿼리
  components/ ← Position, Mesh, Weather, LOD...
  systems/ ← RenderSystem, ChunkSystem, WeatherSystem...
  archetypes/ ← Building, Vehicle, Landmark 템플릿
  assets/ ← 절차적 생성기, 재질 팩토리
  renderer/ ← Three.js 래퍼, 포스트프로세싱

Pros

  • 수만 엔티티도 효율적 처리
  • 날씨→재질 등 크로스커팅이 시스템으로 자연스럽게 해결
  • 프로파일링/최적화 용이

Cons

  • 학습 곡선 높음 (bitECS 패러다임)
  • 현재 코드 대부분 재작성 필요
  • 단순한 작업도 ECS 패턴 강제
C

Hybrid (하이브리드) — 추천

레이어드 모듈 기반 + 대량 엔티티만 ECS. 실용적 타협안.

src/
  engine/ ← 게임 루프, 씬 매니저, 에셋 로더
  world/ ← ChunkManager, LODController, 지형
  landmarks/ ← 클래스 기반 (Bridge, Pyramid...)
  city/ ← 절차적 건물 (InstancedMesh 풀링)
  traffic/ ← ECS로 차량/보트/새 대량 관리
  atmosphere/ ← TimeOfDay, Weather, PostFX 파이프라인
  camera/ ← FlightCamera, CinematicPath, PostFX
  ui/ ← HUD, Minimap (Preact/Solid 경량 UI)

Pros

  • 현재 코드 70%+ 재활용 가능
  • 랜드마크는 직관적 클래스, 대량 엔티티만 ECS
  • 점진적 마이그레이션 가능
  • 필요한 곳에만 복잡도 추가

Cons

  • 두 패러다임 공존 (클래스 + ECS)
  • 경계선 판단이 필요 (언제 ECS? 언제 클래스?)

Why C (Hybrid)?

포트폴리오 데모에서 풀 ECS는 오버엔지니어링입니다. 금문교 같은 랜드마크는 한 번 만들면 끝이라 ECS의 이점이 없고, 오히려 클래스로 캡슐화하는 게 코드 가독성이 좋습니다. 반면 수천 대의 차량, 보행자, 건물 창문 불빛은 InstancedMesh + 경량 ECS가 성능에 결정적입니다. 복잡도는 필요한 곳에만.
